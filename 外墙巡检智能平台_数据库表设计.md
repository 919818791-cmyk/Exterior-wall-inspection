# 外墙巡检智能平台数据库表设计说明

> 适用阶段：MVP 前后端开发  
> 数据库：PostgreSQL  
> 文件存储：MinIO  
> 后端框架：FastAPI  
> 设计目标：支撑项目创建、建筑/立面管理、照片上传、AI 检测任务、算法结果回传、人工审核、报告生成与推送的完整闭环。

---

## 1. 设计原则

本系统建议采用“项目主状态 + 检测任务状态 + 报告状态”分层设计，不建议只用一个 `status` 字段承载全部业务状态。

核心原则：

1. 项目是业务主线，负责表达项目当前处于待检测、检测中、待审核、已审核、已完成等生命周期。
2. 检测任务是算法执行主线，负责表达任务等待、处理中、成功、失败等状态。
3. 报告是交付主线，负责表达报告草稿、已生成、已推送等状态。
4. AI 检测结果和人工审核结果分开存储，避免人工修改覆盖原始 AI 输出。
5. 照片文件不直接存数据库，只保存 MinIO 对象路径、文件信息和业务关联。
6. 正式环境中系统前后端与算法模型部署在同一台服务器；算法 Worker 跟随算法模型部署，建议使用独立 Docker 容器；Worker 不直接连接数据库，只通过后端 API 拉取任务、发送心跳和回传结果。
7. 先满足 MVP 闭环，不提前引入复杂多租户、复杂权限、复杂审批流。

## 2. 核心状态枚举

### 2.1 项目状态 `project.status`

| 状态值 | 中文展示 | 含义 |
| --- | --- | --- |
| draft | 待检测 | 项目已创建，尚未启动 AI 检测，可编辑 |
| detecting | AI检测中 | 已启动 AI 检测，算法处理中，不可编辑 |
| pending_review | 待审核 | AI 检测完成，等待内部人工审核 |
| reviewed | 已审核 | 人工审核完成并已生成报告，等待推送 |
| completed | 已完成 | 报告已推送，用户可查看最终报告 |
| failed | 检测失败 | AI 检测失败，需要重试或人工处理 |

### 2.2 检测任务状态 `detection_task.status`

| 状态值 | 中文展示 | 含义 |
| --- | --- | --- |
| pending | 待处理 | 后端已创建任务，等待算法 Worker 拉取 |
| running | 检测中 | 算法 Worker 已领取任务并开始处理 |
| success | 检测完成 | 算法已成功返回检测结果 |
| failed | 检测失败 | 算法执行失败或结果回传失败 |
| canceled | 已取消 | 任务被人工取消 |

### 2.3 报告状态 `inspection_report.status`

| 状态值 | 中文展示 | 含义 |
| --- | --- | --- |
| draft | 草稿 | 审核过程中或报告尚未正式生成 |
| generated | 已生成 | 已完成人工审核并生成报告 |
| pushed | 已推送 | 报告已推送给用户 |
| revoked | 已撤回 | 已推送报告被撤回或作废 |

### 2.4 审核结果状态 `review_result.status`

| 状态值 | 中文展示 | 含义 |
| --- | --- | --- |
| pending | 待审核 | AI 结果尚未人工确认 |
| confirmed | 已确认 | 人工确认 AI 结果正确 |
| modified | 已修正 | 人工修改了 AI 结果 |
| deleted | 已删除 | 人工删除了误检结果 |
| added | 人工新增 | 人工补充了漏检结果 |

## 4.5 检测时段推荐表 `collection_time_recommendation`

用于保存系统根据项目、计划时间、用户本次选择的立面朝向、天气等条件计算出的推荐采集时间。

| 字段名 | 类型 | 是否必填 | 说明 |
| --- | --- | --- | --- |
| id | UUID | 是 | 主键 |
| project_id | UUID | 是 | 所属项目 |
| building_id | UUID | 否 | 所属建筑 |
| facade_id | UUID | 否 | 所属立面 |
| target_date | DATE | 是 | 计划检测日期 |
| orientation | VARCHAR(32) | 是 | 本次推荐计算选择的立面朝向 |
| recommended_start_time | TIMESTAMP | 是 | 推荐开始时间 |
| recommended_end_time | TIMESTAMP | 是 | 推荐结束时间 |
| score | DECIMAL(5,2) | 否 | 推荐评分 |
| reason | TEXT | 否 | 推荐原因 |
| weather_json | JSONB | 否 | 天气参数，例如温度、风速、风向、降雨、光照等 |
| created_at | TIMESTAMP | 是 | 创建时间 |

### 说明

MVP 阶段可以先用模拟计算逻辑，后续再接入真实天气接口和推荐算法。该表的 `orientation` 是推荐计算输入，不反写到 `facade`。

| 字段名 | 类型 | 是否必填 | 说明 |

| polygon_json | JSONB | 否 | 审核后的多边形 |
| severity | VARCHAR(32) | 否 | 严重程度 |
| area | DECIMAL(10,4) | 否 | 缺陷面积 |

- 人工新增漏检结果时，`ai_result_id` 为空，`status = added`。

# 7. 算法 Worker 对接相关表

算法 Worker 主要使用以下表对应的 API。正式环境中系统前后端与算法模型部署在同一台服务器；Worker 跟随算法模型部署，建议拆成 `algorithm-worker` 与 `algorithm-model` 两个 Docker 容器；Worker 不直接访问数据库。

| 表 | Worker 是否直接访问 | 说明 |
| --- | --- | --- |
| `detection_task` | 否，通过 API | 拉取待处理任务、更新任务状态 |
| `detection_task_photo` | 否，通过 API | 获取任务下的照片列表 |
| `photo` | 否，通过 API | 获取图片下载地址 |
| `ai_detection_result` | 否，通过 API | 回传 AI 检测结果，由后端写入 |

推荐 API 流程：

```text
1. Worker 携带 worker_id、worker_token、model_version 调用 GET /api/algorithm/tasks/next
2. 后端返回一个 pending 任务，并将任务锁定为 running
3. 后端写入 locked_at、worker_heartbeat_at、lease_expires_at、model_version
4. Worker 根据返回的照片列表下载图片
5. Worker 调用同机 Docker 内部网络中的算法模型容器完成推理
6. Worker 定期调用 POST /api/algorithm/tasks/{task_id}/heartbeat
7. Worker 调用 POST /api/algorithm/tasks/{task_id}/results
8. 后端幂等写入 ai_detection_result
9. 后端更新 detection_task.status = success
10. 后端更新 project.status = pending_review
```
