# 第 5 阶段：AI 检测任务、模拟 Worker 与正式算法接入契约

请继续开发 AI 检测任务模块。

本阶段目标：

点击“开始 AI 检测”后，后端创建检测任务；模拟 Worker 可以拉取任务并回传固定 JSON；系统自动把项目状态从 `detecting` 流转到 `pending_review`。

当前开发测试环境不部署真实算法模型，第 5 阶段只实现任务流转、Worker API 契约和模拟 Worker。完成应用功能开发后，系统前端、后端和算法模型会部署在同一台正式服务器；Worker 是算法侧组件，跟随算法模型部署。为了安全，正式部署建议 Worker 和算法模型拆成不同 Docker 容器。

后端实现：

1. 实现启动检测接口：

   * `POST /api/projects/{project_id}/start-detection`

接口职责：

1. 校验项目是否为 `draft`。
2. 校验项目是否已上传照片。
3. 校验是否至少选择一种检测模型。
4. 创建 `detection_task`。
5. 创建 `detection_task_photo`。
6. 将 `project.status` 改为 `detecting`。
7. 将照片状态改为 `detecting`。
8. 返回任务信息。

算法 Worker 接口：

1. `GET /api/algorithm/tasks/next`
2. `POST /api/algorithm/tasks/{task_id}/heartbeat`
3. `POST /api/algorithm/tasks/{task_id}/results`
4. `POST /api/algorithm/tasks/{task_id}/failed`

Worker 接入与容器隔离要求：

1. Worker 请求携带 `worker_id` 和 `worker_token`。
2. Worker 领取任务时上报 `model_version`。
3. 后端返回任务时提供照片下载地址，不要求 Worker 访问数据库。
4. Worker 所在容器只需要能访问后端 API、MinIO 下载地址和内部算法模型地址，不要求后端主动访问 Worker 或模型容器。
5. 模拟 Worker 和真实 Worker 使用同一套 API 契约。
6. Worker 跟随算法模型部署，不放进 Web 后端进程或 Web 后端容器。
7. 正式部署建议拆分 `algorithm-worker` 与 `algorithm-model` 两个容器：Worker 持有后端访问 Token，模型容器只提供内部推理接口。
8. `algorithm-model` 容器不直接访问数据库，不持有 Worker Token，默认不映射公网端口。
9. 后端 API 不直接调用模型容器；真实模型推理由 Worker 通过 Docker 内部网络触发。

任务拉取逻辑：

1. Worker 拉取 `pending` 任务。
2. 后端将任务锁定为 `running`。
3. 写入 `worker_id`、`locked_at`、`started_at`、`worker_heartbeat_at`、`lease_expires_at`、`model_version`。
4. 返回任务关联的照片下载地址、模型类型、高精度配置。

Worker 回传成功逻辑：

1. 保存 `ai_detection_result`。
2. 将 `detection_task.status` 改为 `success`。
3. 将 `project.status` 改为 `pending_review`。
4. 将照片状态改为 `detected`。
5. 写入任务完成时间和结果摘要。

Worker 回传失败逻辑：

1. 将 `detection_task.status` 改为 `failed`。
2. 将 `project.status` 改为 `failed`。
3. 将照片状态改为 `failed`。
4. 保存失败原因。

模拟 Worker：

请写一个简单模拟 Worker，可以用固定 JSON 测试完整流程。

模拟 Worker 需要支持两种运行方式：

1. 在开发机本地运行，用于本机联调。
2. 在 Docker 容器中运行，用于验证正式部署拓扑下的后端 API、MinIO 图片下载和结果回传。

模拟 JSON 至少包含：

* `task_id`
* `project_id`
* `photo_id`
* 缺陷类型
* 置信度
* bbox 坐标
* 严重程度
* 描述
* `model_version`

前端实现：

1. 项目详情页“开始 AI 检测”按钮接入真实接口。
2. 点击后项目状态变为“AI检测中”。
3. `detecting` 状态页面只读，按钮显示“AI检测中，不可点击”。
4. `pending_review` 状态普通用户侧显示“结果审核中”或“报告生成中”，不要暴露内部审核细节。
5. `failed` 状态显示失败原因，并预留“重新检测”。


验收标准：

1. `draft` 项目点击“开始 AI 检测”后变为 `detecting`。
2. Worker 能拉取到 pending 任务。
3. 模拟 Worker 在开发机本地和容器环境中都可以访问后端健康检查接口和任务接口。
4. Worker 能根据任务返回内容下载照片。
5. Worker 回传固定 JSON 后写入 `ai_detection_result`。
6. 项目状态自动变为 `pending_review`。
7. 普通用户看不到审核工作台和 AI 原始审核过程。
8. 文档中明确正式环境为系统前后端与算法模型同服务器部署，Worker 跟随算法模型部署，并推荐 `algorithm-worker` 与 `algorithm-model` 分容器，测试环境不部署真实算法模型。

# 第 6 阶段：审核工作台与缺陷框编辑

请继续开发审核工作台。

本阶段目标：

内部审核人员可以查看 AI 检测结果，在图片上确认、修正、删除、新增缺陷框，并完成审核生成报告记录。

前端路由：

* `/review`
* `/review/projects/:id`

后端接口：

* `GET /api/review/projects`
* `GET /api/review/projects/{project_id}`
* `GET /api/review/projects/{project_id}/results`
* `PUT /api/review/results/{result_id}`
* `POST /api/review/results`
* `DELETE /api/review/results/{result_id}`
* `POST /api/review/projects/{project_id}/complete`

审核工作台列表规则：

1. 不展示 `draft` 项目。
2. `detecting` 项目可显示“AI检测中”，但不能开始审核。
3. `pending_review` 项目显示“开始审核”。
4. `reviewed` 项目显示“预览报告并推送”。
5. `completed` 项目显示“查看报告”。

审核详情功能：

1. 展示项目基础信息。
2. 展示照片列表。
3. 按照片读取 AI 检测结果。
4. 使用 `react-konva` 在图片上渲染缺陷框。
5. 支持缺陷框拖拽。
6. 支持缺陷框缩放。
7. 支持修改缺陷类型。
8. 支持确认 AI 缺陷。
9. 支持删除误检缺陷。
10. 支持新增漏检缺陷。
11. 支持保存审核结果。

数据规则：

1. AI 原始结果不允许被覆盖。
2. 人工审核后的结果保存到 `review_result`。
3. AI 结果确认后，复制一份到 `review_result`，状态为 `confirmed`。
4. AI 结果被修改后，在 `review_result` 保存修改后的结果，状态为 `modified`。
5. AI 结果被删除后，保留一条 `status = deleted` 的审核记录。
6. 人工新增漏检结果时，`ai_result_id` 为空，状态为 `added`。
7. 每次操作写入 `review_operation_log`。

完成审核接口职责：

1. 校验项目是否为 `pending_review`。
2. 校验是否存在有效审核结果。
3. 生成 `inspection_report` 记录。
4. `inspection_report.status = generated`。
5. `project.status = reviewed`。
6. 写入审核操作日志 `generate_report`。

验收标准：

1. 审核工作台能看到 `detecting`、`pending_review`、`reviewed`、`completed` 项目。
2. `pending_review` 项目能进入审核。
3. 图片上能显示 AI 缺陷框。
4. 缺陷框可以确认、修改、删除、新增。
5. 保存人工审核结果不覆盖 AI 原始结果。
6. 完成审核后生成报告记录。
7. 项目状态变为 `reviewed`。

# 第 7 阶段：报告预览、推送与检测报告列表

请继续开发报告模块。

本阶段目标：

内部审核人员可以预览并推送报告；普通用户只能查看已推送的最终报告，并支持 DOCX 报告生成与下载。

报告文件形式调整：

1. 正式交付报告统一为 `.docx` 文件，不再开发 PDF 报告文件、PDF 导出接口或 HTML 转 PDF 流程。
2. 页面内仍保留结构化报告预览，用于审核人员推送前确认和普通用户在线查看。
3. 第 6 阶段已生成的报告记录继续作为报告元数据来源；第 7 阶段需要补充 DOCX 文件生成和文件路径落库。
4. 如果当前数据库模型或迁移中已有 `pdf_bucket`、`pdf_object_key` 字段，第 7 阶段通过新增迁移统一改为 `docx_bucket`、`docx_object_key`，或在尚未进入共享环境时直接重命名模型字段。后续代码不得再新增 PDF 字段和 PDF 相关接口。

后端接口：

* `GET /api/reports`
* `GET /api/reports/{report_id}`
* `POST /api/reports/{report_id}/push`
* `GET /api/reports/{report_id}/docx`

DOCX 生成与下载职责：

1. DOCX 内容基于 `inspection_report.report_data_json` 生成。
2. 生成后的文件上传到 MinIO，并将 `docx_bucket`、`docx_object_key` 写入 `inspection_report`。
3. 下载接口只返回当前报告对应的 DOCX 文件；如果文件尚未生成，可以按当前固化数据即时生成后再返回，或返回明确的待生成错误。
4. 文件名建议使用 `报告编号-报告标题.docx`，注意过滤文件名中的非法字符。

推送报告接口职责：

1. 校验 `inspection_report.status` 是否为 `generated`。
2. 写入 `pushed_at`。
3. 将 `inspection_report.status` 改为 `pushed`。
4. 将 `project.status` 改为 `completed`。
5. 写入 `report_push_log`。

报告内容：

1. 项目基本信息。
2. 委托单位。
3. 检测时间。
4. 建筑与立面信息。
5. 检测模型配置。
6. 缺陷统计。
7. 缺陷明细。
8. 缺陷图片与标注框。
9. 审核结论。
10. 报告生成时间。

报告数据规则：

1. 报告生成时，把关键数据固化到 `inspection_report.report_data_json`。
2. 报告展示优先读取 `report_data_json`。
3. 避免后续项目或审核数据变动影响历史报告。
4. 普通用户只允许看到 `report.status = pushed` 或 `project.status = completed` 的报告。

前端实现：

1. `/reports` 检测报告列表页。
2. `/reports/:id` 报告详情 / 报告预览页。
3. 审核工作台中 `reviewed` 项目可以进入“预览报告并推送”。
4. 推送后项目状态变为 `completed`。
5. 普通用户在检测报告页可以查看最终报告。
6. 报告详情页提供“下载 DOCX”操作，不提供“导出 PDF”。


验收标准：

1. `reviewed` 项目可以预览报告。
2. 推送后 `report.status = pushed`。
3. 推送后 `project.status = completed`。
4. 普通用户报告列表只展示已推送报告。
5. 报告详情能看到项目、建筑、立面、缺陷统计、缺陷明细和标注图片。
6. 可以下载 DOCX 报告文件。
7. 代码、接口、按钮文案和验收流程中不再出现 PDF 导出能力。
